最长回文(Swift版)

马拉车算法简直是太有意思了，将一个暴力时间复杂度为O(n^3 )的问题优化成了一个时间复杂度为O(n)的问题。不得不说，很牛逼，值得记录一次。

问题描述

求给定字符串中的最长回文

什么是回文？类似”abba”、”acdca”这样正序和倒序完全相同的字符串就可以称为回文

方法一：暴力算法

一般碰到回文的问题，可以通过双指针来判断，暴力算法解决
枚举所有子字符串，判断各个子字符串是否是回文，记录最长的那个。
这种算法最简单，但是时间复杂度也是最高的O(n^3 )，并且不能通过Leetcode的测试用例

方法二：动态规划

万能的动态规划解法，只要是涉及到动态规划的问题，就算是比较难的问题了吧。
暴力算法子串的首位遍历，加上每一个子串是否为回文的遍历，组成了三层遍历，也将时间复杂度贡献到了O(n^3 )。如果我们保存每一个子串是否为回文的状态，根据上一个状态来推测当前的状态，这样是否可行呢？当然是可行的！这是一种用空间换时间的方法，在暴力解法的基础上减少了一次遍历，只需要遍历首位子字符串就可以了，成功将时间复杂度降到O(n^3 )。

由于我们想用上一次的状态来判定当前的状态，所以我们定义状态数组dp[i][j]，表示以i开头，j结尾的子字符串是否为回文串。
定义好了状态数组，我们再来定义状态转移方程：

  • dp[i][j] = dp[i + 1][j - 1] && s[i] == s[j]

如果去掉首位两个字符串后的子字符串是回文字符串，且首位字符相等，那当前的子串也一定是回文串。

确定好了状态转移方程，我们再来确定边界条件：
1. 如果i == j，即只有一个字符的时候，这个字符当然是自己的回文
2. 如果i == j - 1 && s[i] == s[j]，即只有两个字符，且两个字符都相等的时候，这两个字符当然是回文
3. 如果i < j，当前子串是不存在的，因为不存在一个子字符串的开始比结尾大

根据状态转移方程，上一个状态的长度肯定比现在的长度要小2，所以我们把长度作为第一层循环，从0到字符串的长度遍历。
第二层循环以开始位置为起点，以结束位置小于字符串长度为终点执行遍历
由于长度在遍历中是递增的，所以后面找到的回文字符串肯定比前面的长，所以保存最后一次找到的回文字符串就可以了。

但是。。。但是。。。在swift下这种方式还是翻车了，依然不能通过上述的测试用例。虽然时间复杂度已经降到了O(n^2 )，空间复杂度升到了O(n^2 )，但还是慢了。
不知道是因为字符串的处理慢，还是其它什么因素导致的有问题。尝试过各种修改，但是在当前思路下，还是没有办法通过测试用例，不得不换一种思路。

方法三：中心扩展算法

连最牛逼的动态规划都已经翻车了，看来动态规划也不是万能的。
中心扩展算法的思想和动态规划其实很类似，当前状态了是可以根据上一个状态以及首尾两个字符是否相等来确定。它枚举所有中心点从中心点向两边扩展，直到无法扩展为止。
需要注意的是扩展过程要同时计算奇回文和偶回文。
中心扩展算法节约了空间复杂度，成功将空间复杂度降到了O(1)，时间复杂度还是O(n^2 )
这个算法比动态规划牛逼，能优化一点算一点，提交！
MMP，还是翻车了。这TM是什么神仙测试用例。不自觉的提出了我50米的大刀～

方法四：马拉车算法

但凡上面的某一个算法能通过测试用例，我也不至于走到这一步，都TM是力扣逼的。它对效率的追求还真是永无止境，李笑来说过，“什么是牛逼的人，有拿第一的执念，有拿第一的经验，有多次拿第一的经验和总结”，牛逼就是从执念开始的。
为了踏出牛逼的第一步，只能研究这个马拉车算法。

预处理

动态规划和中心扩展方案中都把奇回文和偶回文单独拿出来处理了一遍，麻烦！
让我们来处理一下，在每个字符的两边都加上一个相同的特殊符号，这里常用#号。
处理后的字符长度变为了2*n + 1，始终为奇数，那我们就只需要考虑奇数回文的问题了。

计算最长回文串的长度

我们用一个字符串长度的数组了保存每一个中心点的最大扩展半径，最大值就是最长回文串的中心点了。
我们以字符串cabbaf为例，处理后的字符串是#c#a#b#b#a#f#，以每一位字符为中心点，向两端扩展的最大回文半径对应如下：

第6个字符的回文半径是最大的R=5，回文串为#a#b#b#a#，回文串长度为length=9，实际字符长度为n=4。
由于在预处理时在所有字符前后都加了#，所以length = 2 * n + 1。而R = (length + 1) / 2
得出实际半价和最大回文半径的关系为：n = R - 1即：n = 5 - 1 => n = 4

最长回文串起始索引

算出了最大回文串的长度，我们还需要确定起始索引，才能确定唯一的最长回文字符串。
还是以cabbaf字符串为例，原始字符的索引i和预处理后字符锁索引I的关系如下：I = i * 2 + 1
根据中心点和最大半径可知预处理后的起始索引 startIndex_I = center_I - p[center_I] + 1
那么startIndex_i = (center_I - p[center_I]) / 2。即：startIndex_i = (6 - 5) / 2 => startIndex_i = 1/2

再带入一个奇回文aba看看，预处理之后是#a#b#a#，p[3]=4，代入公式后变成了 startIndex_i = (3 - 4) / 2 => startIndex_i = - 1/2
还搞成负数了～ 仔细想想，是原来0索引前面一个数，硬要算的话，- 1/2其实是合理的，但对我来说0才是正确的。

怎么办呢？看一眼题解[奸笑]

原来在首位还加了特殊的字符，这个特殊字符的目的就是为了防止数组越界，随便什么字符都可以，我就用正则表达式的开始和结尾来作为特殊字符吧。
由于前面两个字符都是占位用的，所以原始字符的索引i和预处理后字符锁索引I的关系：i = I / 2 - 1
那么startIndex_i = (center_I - p[center_I]) / 2 - 1/2，向下取整。
如果是偶回文，中心点一定在特殊符号#上，所以center_I一定是奇数，回文也一定是奇数，奇数减奇数是偶数，所以(center_I - p[center_I]) / 2一定是整数，那么可以直接舍掉-1/2
如果是奇回文，中心点一定在字符上，center_I一定是偶数，回文一定是偶数，所以(center_I - p[center_I]) / 2一定是整数，也可以直接舍掉-1/2
那么startIndex_i = (center_I - p[center_I]) / 2

数组p

进行到这里，基本上就已经结束了，我们就开始～ 等等～
从始至终，我们只提到了怎么用p，但是我们对p的创建只字未提，我们来看看如何创建p。
创建方式其实和中心扩展算法的思路很像，遍历中心点，左右扩展。
因为我们遍历的中心点的左右是对称的，所以当前最大回文中心点右侧如果有一个回文串，那左侧也一定有一个相同的回文串，这里就不用再计算了。

值得注意的是右侧回文串的范围超过了最大回文串的范围时，我们只取范围内的值，范围外的值还是需要通过中心扩展来计算。
